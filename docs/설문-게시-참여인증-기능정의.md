# 설문 게시 · 참여 인증 플로우 — 기능 정의 (Goal 인풋)

> 어드민(veyor-admin)의 **설문 등록·게시 관리**와, 메인 앱(veyor-app)의 **설문 참여→완료 인증 무결성** 플로우를 정의한다.
> 관련 레포: `veyor-admin`(운영자 어드민) · `veyor-app`(백엔드 NestJS + 소비자 client).

---

## 1. 목적 / 배경

- 운영자가 접수·수기 등록한 설문을 **검토 → 게시**하면, 정해진 날짜에 앱 "오늘의 설문"으로 노출된다.
- 사용자는 설문 참여 시 외부 설문 페이지로 이동했다가 **우리 인증 주소로 복귀**해 완료를 인증하고 리워드를 받는다.
- **핵심 무결성 요구**: 외부 인증 주소(`/surveys/:id/complete`)는 고정·공개이므로, "우리 앱의 참여 버튼을 눌러 시작한 사용자"만 완료 인증이 되도록 막아야 한다. (직접 URL 접속만으로 리워드 취득하는 어뷰징 차단)

---

## 2. 전체 플로우

```
[메인 앱]  "설문 참여" 클릭
   │  ① POST /surveys/:id/start   ← "이 유저가 버튼을 눌러 시작했다" 기록 (status=started)
   ▼
[외부 설문 페이지] (survey.external_url) ... 응답 작성 ...
   │  설문 마지막에 운영자가 붙여둔 우리 인증 주소
   │  → https://bsg.io.kr/surveys/:id/complete
   ▼
[우리 프론트 /surveys/:id/complete]   (로그인 세션으로 유저 식별)
   │  ② POST /surveys/:id/complete
   ▼
[백엔드] 이 유저의 start 기록 확인
   ├─ 있음 → status=completed 전이 + 리워드(pending) 생성 ✅
   └─ 없음 → 거부 ("설문을 시작하지 않았습니다") ❌
```

- 인증 주소는 설문마다 **고정**(`/surveys/:id/complete`). 복귀 시 **로그인 세션으로 유저를 식별**하고, 그 유저의 **start 기록**을 게이트로 사용한다 → 별도 유저별 토큰 없이도 "버튼 거쳐 들어온 사람"만 인증.
- 이 인증 주소는 어드민이 **게시 시 발급**하며, 운영자가 외부 설문 마지막 페이지에 붙여넣는다.

## 3. 데이터 모델 변경 (veyor-app)

### 3.1 `participations` — 'started' 상태 도입

| 컬럼 | 변경 | 비고 |
| --- | --- | --- |
| `status` | `check (status in ('started','completed'))`, default `'started'` | 기존엔 `'completed'`만 허용 |
| `started_at` | **신규** `timestamptz not null default now()` | 참여 버튼 누른 시점 |
| `completed_at` | **nullable로 변경** (완료 시 채움) | 기존 NOT NULL |

- `UNIQUE(user_id, survey_id)` 유지 → 한 행이 `started → completed`로 **전이**.
- 재시작은 멱등(이미 행 있으면 그대로), 재완료는 409.

### 3.2 노출 게이트 (`surveys`)

- 앱 노출은 `is_published=true` **AND** `approval_status='approved'` **AND** 기간(`opens_at ≤ now ≤ expires_at`)을 모두 만족해야 한다.
- (현재 버그: 노출 후보 조회가 시간창만 보고 `is_published`/`approval_status`를 보지 않음 → 반드시 게이트 추가.)

### 3.3 마이그레이션 드리프트 복구

- `survey_intakes`·`survey_operations` 테이블이 **DB에는 존재하나 veyor-app 마이그레이션 파일이 없음**(소스 관리 밖).
- 해당 스키마(접수/운영 분리 + 위 변경)를 veyor-app 마이그레이션으로 **복구·커밋**해 재현 가능하게 한다.

## 4. 백엔드 API (veyor-app)

| 메서드 | 경로 | 변경 | 설명 |
| --- | --- | --- | --- |
| POST | `/surveys/:surveyId/start` | **신규** | 노출중(게시·승인·기간내) 검증 후 `participation(status='started')` 생성(멱등) |
| POST | `/surveys/:surveyId/complete` | **수정** | **start 기록 존재를 게이트**로 추가. 있으면 `completed` 전이 + 리워드(pending) 생성, 없으면 거부 |
| GET | `/surveys/today` | 수정 | 노출 게이트(3.2) 반영 |

**에러 정책(추가)**

| 상황 | 코드 | 메시지 |
| --- | --- | --- |
| start 없이 complete 호출 | 400/403 | "설문을 완료할 수 없습니다." (시작 여부는 노출하지 않고 일반 메시지로만 안내) |
| (기존) 이미 참여 완료 | 409 | "이미 참여한 설문입니다." |
| (기존) 만료된 설문 | 410 | "참여 기간이 지난 설문입니다." |

## 5. 어드민 (veyor-admin)

### 5.1 설문 등록 → 누적

- 공개 접수 폼(`/submit`) 또는 수기 등록(`/surveys/new`)으로 등록 시 `is_published=false`로 저장 → **"모든 설문" 뷰에 누적**.

### 5.2 게시 = 단일 날짜 스케줄

- 게시 폼의 노출 기간을 **범위(시작/종료 datetime 2개) → 단일 날짜(date 1개)** 로 변경.
- "게시" 시 서버가 계산:
  - `opens_at = {게시일}T10:00:00+09:00` (**오전 10시 KST**)
  - `expires_at = 익일 10:00 KST` (**1일 노출**)
  - `is_published=true`, `approval_status='approved'`
- 기본 게시일 = 접수 시 입력된 `requested_publish_date`.
- 게시 취소(`unpublish`)도 제공.

### 5.3 두 개의 관리 뷰

| 뷰 | 내용 |
| --- | --- |
| **모든 설문** | 전체 누적(접수+수기). **정렬**(헤더 클릭: 게시일·금액·생성일·승인상태 등) + **필터**(승인/출처/게시여부/정산/게시일 기간/검색) |
| **등록된 설문** | 게시된 것만. 게시일·노출기간 + **완료 인증 링크(복사 버튼)** 표시. 게시 취소. |

- 구성: 탭 전환(`모든 설문` / `등록된 설문`).

### 5.4 완료 인증 링크 발급

- 형식: `https://bsg.io.kr/surveys/{surveyId}/complete` (베이스 도메인은 env로 구성).
- "등록된 설문" 뷰에 노출 + 복사 버튼. 운영자가 외부 설문 마지막에 붙여넣는다.

## 6. 소비자 프론트 (veyor-app/apps/client)

> **별도 담당 영역(본 문서 범위 외).** 완료 페이지는 이미 구현된 것으로 파악.
> 본 문서는 백엔드 API(`start`/`complete`)와 그 계약만 정의하며, client는 이 계약에 맞춰 연동한다.

---

## 7. 파급효과 (반드시 점검)

- `started` 행이 생기므로 **연속참여(streak)·"참여함" 표시·누적 집계는 `status='completed'`만** 카운트하도록 필터 추가. (대상: `getStreak`, `getToday`의 `participated` 플래그 등)

## 8. 범위 / 레포 구분

| Phase | 레포 | 작업 |
| --- | --- | --- |
| 1 | veyor-app(백엔드) | 마이그레이션 복구 · 노출 게이트 · `participations` 'started' · `start`/`complete` 무결성 |
| 2·3 | veyor-admin | 단일날짜 게시 · 두 뷰 · 정렬/필터 · 완료 링크 발급 |
| — | veyor-app(client) | **별도 담당(범위 외).** 백엔드 `start`/`complete` 계약에 맞춰 연동 |

---

## 9. 비고 / 한계

- 본 무결성 장치는 **"우리 앱 버튼으로 시작 → 복귀"** 까지 보장한다. 외부 설문을 실제로 끝냈는지는 외부 도구라 검증 불가(원천적 한계).
- 강화 옵션: start 시 1회용 토큰 발급 → 복귀 URL에 부착·검증. 단, 운영자가 붙이는 링크가 **유저별**이 되어야 해서 고정 링크 발급과 상충 → 현재는 `start기록 + 로그인 + UNIQUE(1회)` 조합으로 직접접속 어뷰징을 차단(토큰은 선택적 강화책).
